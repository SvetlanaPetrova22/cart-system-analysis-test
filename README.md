2. Исправленная версия ТЗ
1) Пользователь может добавить в корзину от 1 до 10 единиц одного товара.
2) Пользователь может изменять количество товара в корзине в диапазоне от 1 до 10 единиц. Для удаления
товара используется отдельная кнопка "Удалить".
3) В корзине может находиться не более 5 различных товаров (уникальных позиций).
4) Суммарное количество всех товаров в корзине не может превышать 20 единиц.
5) При попытке добавить товар система выполняет проверки:
   не превышено ли количество данного товара (<=10);
   не превышено ли общее количество товаров (<=20);
   не превышено ли количество различных товаров (<=5);
6) В случае превышения лимитов система отображает соответствующее сообщение:
   • "Превышено максимальное количество единиц товара (10)";
   • "Превышено общее количество товаров в корзине (20)";
   • "Превышено количество различных товаров в корзине (5)".
7) Цена товара фиксируется на момент добавления в корзину и не изменяется, 
независимо от изменений в каталоге.
8) На странице корзины отображается: список товаров, количество, цена за единицу (зафиксированная),
стоимость позиции (количество*цена), итоговая стоимость корзины.
9) В корзине может отображаться рекламный блок.
10) Рекламный блок отображается в будние дни (понедельник-пятница): 
   • Утро: 08:00-11:59;
   • Вечер: 18:00-21:59.
По времени пользователя.

3. АРХИТЕКТУРА. (Схема находится в репозитории в формате .png)
Источники триггеров - это микросервисы, которые знают, когда что-то произошло:
• Order Service публикует события при смене статуса или отмене заказа.
• Cart Service фиксирует брошенные корзины (товар лежит N часов без оформления).
• Marketing Service инициирует массовые рекламные рассылки.
• Admin Panel позволяет отправить пуш вручную — например, по конкретному пользователю.

Message Broker (Kafka / RabbitMQ) — шина событий между сервисами. 
Микросервисы не знают друг о друге: Order Service просто публикует событие "order.cancelled", 
а дальше это не его забота. Это развязка зависимостей.
Notification Service — главный исполнитель. Внутри него:
• Event Listener — подписан на все нужные топики в брокере.
• Rate Limiter & Dedup Filter — защита от спама: не шлёт 10 одинаковых пушей подряд, 
  не бомбит пользователя ночью.
• Template Manager — хранит тексты и форматы пушей по типу события.
• User Preferences — проверяет, не отписался ли пользователь от данного типа уведомлений.
• Notification Scheduler — поддержка отложенных пушей (например, «напомнить через 2 часа»).
• Notifications DB — история отправок и статусы (доставлено / прочитано / ошибка).

Device Registry (Token Store) — хранилище push-токенов устройств. Мобильное приложение при запуске 
регистрирует токен, который потом используется для адресной доставки. 
Push Providers — стандартные шлюзы платформ:
• FCM (Firebase Cloud Messaging) — для Android.
• APNs (Apple Push Notification service) — для iOS.

Ключевые решения:
1) Один Notification Service для всех типов пушей - проще контролировать частоту 
и настройки пользователя.
2) Асинхронность через брокер - другие сервисы не ждут ответа и не падают, если пуш не ушёл.
3) Device Token хранится отдельно - при обновлении приложения или переустановке 
токен обновляется в одном месте.
